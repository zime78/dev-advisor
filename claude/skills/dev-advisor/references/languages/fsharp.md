# F\#

원천: `docs/PROGRAMMING_LANGUAGES_2026.md`, 기준일 2026-05-13. 최신 순위와 시장 지표는 답변 시점에 웹으로 확인한다.

## 핵심 판단

.NET 생태계에서 함수형 도메인 모델링을 도입하려면 좋은 선택지다. F\# 도입 판단은 문법 선호보다 실행 환경, 기존 자산, 운영 인력의 숙련도에 더 크게 좌우된다. 제품의 주 경로가 .NET 함수형 개발, 데이터 처리, 도메인 모델링이고, 다음 제약을 통제할 수 있다면 우선 검토할 만하다: .NET 내부에서도 C#보다 채택률이 낮다.

## 사용처

.NET 함수형 개발, 데이터 처리, 도메인 모델링. 실무에서는 이 범위를 넘어 모든 문제를 해결하는 범용 선택지로 보기보다, F\# 생태계가 이미 강한 플랫폼과 도구 체인에 맞춰 채택하는 편이 안정적이다. 신규 프로젝트에서는 장기 유지보수 인력과 배포 대상의 제약을 함께 확인해야 한다.

## 특징

OCaml 계열 문법과 .NET 생태계를 결합했다. 코드 작성 모델, 빌드 방식, 런타임 제약이 프로젝트 구조에 직접 영향을 주므로 초기 설계 단계에서 테스트 전략과 패키지 관리 방식을 같이 정해야 한다.

## 장점

C#과 함께 쓰면서 함수형 모델링을 도입할 수 있다. 특히 기존 생태계의 표준 도구를 그대로 활용할 수 있을 때 생산성과 운영 예측 가능성이 높다. 팀이 관용구와 디버깅 방식을 익히면 같은 기능을 더 적은 접착 코드로 구현할 수 있다.

## 제약

.NET 내부에서도 C#보다 채택률이 낮다. 이 제약은 작은 실험에서는 드러나지 않다가 배포, 성능, 보안 감사, 장기 유지보수 단계에서 비용으로 나타나는 경우가 많다. 도입 전에는 대체 언어와 운영 모델을 같이 비교해야 한다.

## 적합한 프로젝트

- 데이터 처리, 금융/도메인 모델링, .NET 백엔드 일부, 스크립팅.
- C# 자산과 함께 함수형 스타일을 쓰려는 팀.
- F\# 표준 도구와 런타임을 팀이 직접 운영할 수 있고, 코드 리뷰에서 언어별 관용구를 일관되게 적용할 수 있는 프로젝트.

## 부적합하거나 주의할 프로젝트

- 조직 표준이 C#이고 F# 운영자가 거의 없는 신규 서비스.
- UI/플랫폼 SDK 자료가 C# 중심인 영역.
- F\# 경험자가 없는데 핵심 장애 대응, 성능 튜닝, 릴리스 자동화까지 동시에 요구되는 프로젝트.

## 학습 난이도와 선행 지식

함수형 사고, pattern matching, discriminated union, computation expression, .NET interop을 익혀야 한다. 입문 단계에서는 문법보다 작은 프로그램을 빌드, 테스트, 포맷, 배포하는 전체 흐름을 먼저 익히는 것이 좋다. 실무 투입 전에는 오류 처리, 의존성 관리, 표준 라이브러리 사용법을 팀 규칙으로 정리해야 한다.

## 운영/배포 관점

배포와 런타임은 .NET 기준을 따르며, C# 프로젝트와의 인터페이스 경계를 명확히 해야 한다. 재현 가능한 빌드, 의존성 잠금, 런타임 버전 고정, 관측 로그 포맷을 초기에 정하지 않으면 운영 환경 차이로 인한 결함이 늘어난다.

## 타입/런타임 특성

강한 정적 타입과 타입 추론을 제공하는 .NET 기반 ML 계열 언어다. 이 특성은 API 경계, 테스트 범위, 병렬성 모델, 장애 격리 방식에 영향을 준다. 언어의 타입 시스템이 보장하지 않는 부분은 정적 분석, 테스트, 코드 리뷰 규칙으로 보완해야 한다.

## 실사용 예제

다음 예제는 이 언어를 단순 출력이 아니라 실제 업무 코드에서 자주 나오는 데이터 집계, 자동화, 런타임 제어, 또는 플랫폼 특화 작업에 적용하는 형태로 보여준다.

```fsharp
type Order = { Region: string; Amount: decimal; Paid: bool }

let orders =
    [ { Region = "apac"; Amount = 120000M; Paid = true }
      { Region = "emea"; Amount = 90000M; Paid = true }
      { Region = "apac"; Amount = 30000M; Paid = false } ]

let report =
    orders
    |> List.filter _.Paid
    |> List.groupBy _.Region
    |> List.map (fun (region, rows) -> region, rows |> List.sumBy _.Amount)

report
|> List.sortByDescending snd
|> List.iter (fun (region, revenue) -> printfn "%s: %M" region revenue)
```

- 제품의 핵심 경로가 .NET 함수형 개발, 데이터 처리, 도메인 모델링이고, 팀이 F\#의 표준 빌드·테스트·배포 흐름을 문서화할 수 있을 때 도입 우선순위를 높인다.
- 다음 강점이 일정, 품질, 운영 비용 중 하나를 명확히 낮출 때 실무 채택 근거가 충분하다: C#과 함께 쓰면서 함수형 모델링을 도입할 수 있다.
- 다음 제약이 핵심 리스크라면 파일럿 구현, 성능 측정, 장애 대응 연습을 거친 뒤 본격 도입한다: .NET 내부에서도 C#보다 채택률이 낮다.
