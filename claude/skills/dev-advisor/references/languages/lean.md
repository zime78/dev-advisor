# Lean 4

원천: `docs/PROGRAMMING_LANGUAGES_2026.md`, 기준일 2026-05-13. 최신 순위와 시장 지표는 답변 시점에 웹으로 확인한다.

## 핵심 판단

의존 타입(dependent types) 기반 형식 증명 언어이자 함수형 프로그래밍 언어. 수학 정리 증명(mathlib)과 안전성 증명이 필요한 소프트웨어에 검토한다. Lean 4 도입 판단은 문법 선호보다 실행 환경, 기존 자산, 운영 인력의 숙련도에 더 크게 좌우된다. 제품의 주 경로가 수학 정리의 형식 증명, 안전 크리티컬 소프트웨어 사양 검증, 컴파일러·언어 이론 연구, 함수형 프로그래밍 교육(고급)이고, 다음 제약을 통제할 수 있다면 우선 검토할 만하다: .

## 사용처

수학 정리의 형식 증명, 안전 크리티컬 소프트웨어 사양 검증, 컴파일러·언어 이론 연구, 함수형 프로그래밍 교육(고급). 실무에서는 이 범위를 넘어 모든 문제를 해결하는 범용 선택지로 보기보다, Lean 4 생태계가 이미 강한 플랫폼과 도구 체인에 맞춰 채택하는 편이 안정적이다. 신규 프로젝트에서는 장기 유지보수 인력과 배포 대상의 제약을 함께 확인해야 한다.

## 특징

의존 타입(명제가 곧 타입), 전술(tactic) 기반 증명, 함수형 프로그래밍 언어로서의 완전한 기능, mathlib 수학 라이브러리. 코드 작성 모델, 빌드 방식, 런타임 제약이 프로젝트 구조에 직접 영향을 주므로 초기 설계 단계에서 테스트 전략과 패키지 관리 방식을 같이 정해야 한다.

## 장점

. 특히 기존 생태계의 표준 도구를 그대로 활용할 수 있을 때 생산성과 운영 예측 가능성이 높다. 팀이 관용구와 디버깅 방식을 익히면 같은 기능을 더 적은 접착 코드로 구현할 수 있다.

## 제약

. 이 제약은 작은 실험에서는 드러나지 않다가 배포, 성능, 보안 감사, 장기 유지보수 단계에서 비용으로 나타나는 경우가 많다. 도입 전에는 대체 언어와 운영 모델을 같이 비교해야 한다.

## 적합한 프로젝트

- 수학 정리의 기계 검증 (학술·연구)
- 암호화 프로토콜, 프로토콜 명세의 형식 검증
- 컴파일러 정확성 증명, 타입 이론 연구
- Lean 4 표준 도구와 런타임을 팀이 직접 운영할 수 있고, 코드 리뷰에서 언어별 관용구를 일관되게 적용할 수 있는 프로젝트.

## 부적합하거나 주의할 프로젝트

- 일반 웹/앱/데이터 서비스 — Python, Go, TypeScript가 적합
- 증명 없이 함수형 프로그래밍만 원하는 경우 — Haskell, Scala, Elixir가 더 실용적

## 학습 난이도와 선행 지식

수학적 증명 감각(귀납법, 집합론)과 타입 이론(Curry-Howard, 의존 타입) 선행 지식이 필수다. Haskell 등 강한 타입 함수형 언어 경험이 있으면 함수형 부분의 장벽이 낮아진다. "Theorem Proving in Lean 4"와 "Functional Programming in Lean" 두 공식 교재가 있다. 입문 단계에서는 문법보다 작은 프로그램을 빌드, 테스트, 포맷, 배포하는 전체 흐름을 먼저 익히는 것이 좋다. 실무 투입 전에는 오류 처리, 의존성 관리, 표준 라이브러리 사용법을 팀 규칙으로 정리해야 한다.

## 운영/배포 관점

증명 검증 목적의 오프라인 빌드가 주용도. 함수형 언어로서 실행 바이너리 생성 가능하지만 프로덕션 서비스 배포 사례는 드물다. lake로 의존성 관리. 재현 가능한 빌드, 의존성 잠금, 런타임 버전 고정, 관측 로그 포맷을 초기에 정하지 않으면 운영 환경 차이로 인한 결함이 늘어난다.

## 타입/런타임 특성

의존 타입 정적 검사, 컴파일 타임 증명 확인, 런타임은 순수 함수형. 계산 전략은 강한 정규화(strong normalization) 전제. 부분 함수는 명시적 처리 필요. 이 특성은 API 경계, 테스트 범위, 병렬성 모델, 장애 격리 방식에 영향을 준다. 언어의 타입 시스템이 보장하지 않는 부분은 정적 분석, 테스트, 코드 리뷰 규칙으로 보완해야 한다.

## 실사용 예제

다음 예제는 이 언어를 단순 출력이 아니라 실제 업무 코드에서 자주 나오는 데이터 집계, 자동화, 런타임 제어, 또는 플랫폼 특화 작업에 적용하는 형태로 보여준다.

```lean
structure Order where
  region : String
  amount : Nat
  paid : Bool
deriving Repr

def paidRevenue (orders : List Order) : Nat :=
  orders.foldl
    (fun total order =>
      if order.paid then total + order.amount else total)
    0

#eval paidRevenue [
  { region := "apac", amount := 120000, paid := true },
  { region := "emea", amount := 90000, paid := true }
]
```
