# PL/SQL

원천: `docs/PROGRAMMING_LANGUAGES_2026.md`, 기준일 2026-05-13. 최신 순위와 시장 지표는 답변 시점에 웹으로 확인한다.

## 핵심 판단

Oracle DB 내부에서 데이터 가까운 업무 로직을 수행해야 할 때 적합하다. PL/SQL 도입 판단은 문법 선호보다 실행 환경, 기존 자산, 운영 인력의 숙련도에 더 크게 좌우된다. 제품의 주 경로가 Oracle Database 저장 프로시저, 배치, 업무 로직이고, 다음 제약을 통제할 수 있다면 우선 검토할 만하다: Oracle 종속성이 강하고 로직이 DB에 과도하게 모이면 관리가 어렵다.

## 사용처

Oracle Database 저장 프로시저, 배치, 업무 로직. 실무에서는 이 범위를 넘어 모든 문제를 해결하는 범용 선택지로 보기보다, PL/SQL 생태계가 이미 강한 플랫폼과 도구 체인에 맞춰 채택하는 편이 안정적이다. 신규 프로젝트에서는 장기 유지보수 인력과 배포 대상의 제약을 함께 확인해야 한다.

## 특징

SQL에 절차형 프로그래밍을 더한 Oracle 언어. 코드 작성 모델, 빌드 방식, 런타임 제약이 프로젝트 구조에 직접 영향을 주므로 초기 설계 단계에서 테스트 전략과 패키지 관리 방식을 같이 정해야 한다.

## 장점

데이터 가까이에서 복잡한 처리를 빠르게 수행할 수 있다. 특히 기존 생태계의 표준 도구를 그대로 활용할 수 있을 때 생산성과 운영 예측 가능성이 높다. 팀이 관용구와 디버깅 방식을 익히면 같은 기능을 더 적은 접착 코드로 구현할 수 있다.

## 제약

Oracle 종속성이 강하고 로직이 DB에 과도하게 모이면 관리가 어렵다. 이 제약은 작은 실험에서는 드러나지 않다가 배포, 성능, 보안 감사, 장기 유지보수 단계에서 비용으로 나타나는 경우가 많다. 도입 전에는 대체 언어와 운영 모델을 같이 비교해야 한다.

## 적합한 프로젝트

- Oracle 저장 프로시저, 배치 처리, 데이터 검증, 트리거/패키지 기반 업무 로직.
- DB 내부 처리로 네트워크 왕복과 데이터 이동을 줄여야 하는 경우.
- PL/SQL 표준 도구와 런타임을 팀이 직접 운영할 수 있고, 코드 리뷰에서 언어별 관용구를 일관되게 적용할 수 있는 프로젝트.

## 부적합하거나 주의할 프로젝트

- 애플리케이션 로직 대부분을 DB에 숨겨 배포/테스트가 어려워지는 설계.
- Oracle 종속성을 낮춰야 하는 제품.
- PL/SQL 경험자가 없는데 핵심 장애 대응, 성능 튜닝, 릴리스 자동화까지 동시에 요구되는 프로젝트.

## 학습 난이도와 선행 지식

SQL, 트랜잭션, 패키지, 커서, 예외 처리, 실행 계획을 함께 익혀야 한다. 입문 단계에서는 문법보다 작은 프로그램을 빌드, 테스트, 포맷, 배포하는 전체 흐름을 먼저 익히는 것이 좋다. 실무 투입 전에는 오류 처리, 의존성 관리, 표준 라이브러리 사용법을 팀 규칙으로 정리해야 한다.

## 운영/배포 관점

DB 배포, 권한, lock, 성능 계획, 백업/복구, schema migration을 엄격히 관리해야 한다. 재현 가능한 빌드, 의존성 잠금, 런타임 버전 고정, 관측 로그 포맷을 초기에 정하지 않으면 운영 환경 차이로 인한 결함이 늘어난다.

## 타입/런타임 특성

Oracle SQL 확장 절차형 언어이며 DB 엔진의 타입/트랜잭션 모델에 묶인다. 이 특성은 API 경계, 테스트 범위, 병렬성 모델, 장애 격리 방식에 영향을 준다. 언어의 타입 시스템이 보장하지 않는 부분은 정적 분석, 테스트, 코드 리뷰 규칙으로 보완해야 한다.

## 실사용 예제

다음 예제는 이 언어를 단순 출력이 아니라 실제 업무 코드에서 자주 나오는 데이터 집계, 자동화, 런타임 제어, 또는 플랫폼 특화 작업에 적용하는 형태로 보여준다.

```sql
DECLARE
  v_total NUMBER := 0;
BEGIN
  SELECT SUM(amount)
    INTO v_total
    FROM orders
   WHERE status = 'PAID'
     AND paid_at >= ADD_MONTHS(TRUNC(SYSDATE, 'MM'), -1);

  INSERT INTO revenue_audit(run_at, total_amount)
  VALUES (SYSTIMESTAMP, v_total);

  COMMIT;
  DBMS_OUTPUT.PUT_LINE('monthly revenue=' || v_total);
EXCEPTION
  WHEN OTHERS THEN
    ROLLBACK;
    RAISE;
END;
```
