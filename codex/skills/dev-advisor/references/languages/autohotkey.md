# AutoHotkey

원천: `docs/PROGRAMMING_LANGUAGES_2026.md`, 기준일 2026-05-13. 최신 순위와 시장 지표는 답변 시점에 웹으로 확인한다.

## 핵심 판단

Windows 개인 자동화와 단축키 매크로에는 빠르지만 대규모 소프트웨어용은 아니다. AutoHotkey 도입 판단은 문법 선호보다 실행 환경, 기존 자산, 운영 인력의 숙련도에 더 크게 좌우된다. 제품의 주 경로가 Windows 키보드/마우스 자동화, 단축키 매크로이고, 다음 제약을 통제할 수 있다면 우선 검토할 만하다: 대규모 소프트웨어 개발에는 적합하지 않다.

## 사용처

Windows 키보드/마우스 자동화, 단축키 매크로. 실무에서는 이 범위를 넘어 모든 문제를 해결하는 범용 선택지로 보기보다, AutoHotkey 생태계가 이미 강한 플랫폼과 도구 체인에 맞춰 채택하는 편이 안정적이다. 신규 프로젝트에서는 장기 유지보수 인력과 배포 대상의 제약을 함께 확인해야 한다.

## 특징

데스크톱 자동화에 특화된 스크립트 언어. 코드 작성 모델, 빌드 방식, 런타임 제약이 프로젝트 구조에 직접 영향을 주므로 초기 설계 단계에서 테스트 전략과 패키지 관리 방식을 같이 정해야 한다.

## 장점

개인 업무 자동화와 단축키 구성에 빠르다. 특히 기존 생태계의 표준 도구를 그대로 활용할 수 있을 때 생산성과 운영 예측 가능성이 높다. 팀이 관용구와 디버깅 방식을 익히면 같은 기능을 더 적은 접착 코드로 구현할 수 있다.

## 제약

대규모 소프트웨어 개발에는 적합하지 않다. 이 제약은 작은 실험에서는 드러나지 않다가 배포, 성능, 보안 감사, 장기 유지보수 단계에서 비용으로 나타나는 경우가 많다. 도입 전에는 대체 언어와 운영 모델을 같이 비교해야 한다.

## 적합한 프로젝트

- Windows 단축키, 마우스/키보드 자동화, 개인 업무 매크로, 레거시 앱 조작.
- GUI 자동화로 반복 작업을 빠르게 줄이는 개인/소규모 환경.
- AutoHotkey 표준 도구와 런타임을 팀이 직접 운영할 수 있고, 코드 리뷰에서 언어별 관용구를 일관되게 적용할 수 있는 프로젝트.

## 부적합하거나 주의할 프로젝트

- 보안/감사/배포가 중요한 조직 표준 자동화.
- 복잡한 비즈니스 로직이나 크로스플랫폼 앱.
- AutoHotkey 경험자가 없는데 핵심 장애 대응, 성능 튜닝, 릴리스 자동화까지 동시에 요구되는 프로젝트.

## 학습 난이도와 선행 지식

hotkey, hotstring, window spy, send/click, v1/v2 문법 차이를 익혀야 한다. 입문 단계에서는 문법보다 작은 프로그램을 빌드, 테스트, 포맷, 배포하는 전체 흐름을 먼저 익히는 것이 좋다. 실무 투입 전에는 오류 처리, 의존성 관리, 표준 라이브러리 사용법을 팀 규칙으로 정리해야 한다.

## 운영/배포 관점

사용자 PC 권한, 백그라운드 실행, 보안 정책, 키 입력 부작용, 버전 차이를 관리해야 한다. 재현 가능한 빌드, 의존성 잠금, 런타임 버전 고정, 관측 로그 포맷을 초기에 정하지 않으면 운영 환경 차이로 인한 결함이 늘어난다.

## 타입/런타임 특성

Windows 데스크톱 자동화에 특화된 동적 스크립팅 환경이다. 이 특성은 API 경계, 테스트 범위, 병렬성 모델, 장애 격리 방식에 영향을 준다. 언어의 타입 시스템이 보장하지 않는 부분은 정적 분석, 테스트, 코드 리뷰 규칙으로 보완해야 한다.

## 실사용 예제

다음 예제는 이 언어를 단순 출력이 아니라 실제 업무 코드에서 자주 나오는 데이터 집계, 자동화, 런타임 제어, 또는 플랫폼 특화 작업에 적용하는 형태로 보여준다.

```autohotkey
#Requires AutoHotkey v2.0

orders := [
  Map("region", "apac", "amount", 120000, "paid", true),
  Map("region", "emea", "amount", 90000, "paid", true),
  Map("region", "apac", "amount", 30000, "paid", false)
]

report := Map()
for order in orders {
  if !order["paid"]
    continue
  region := order["region"]
  report[region] := report.Get(region, 0) + order["amount"]
}

MsgBox "APAC revenue: " report.Get("apac", 0)
```

## 추천 학습/도입 상황

- 제품의 핵심 경로가 Windows 키보드/마우스 자동화, 단축키 매크로이고, 팀이 AutoHotkey의 표준 빌드·테스트·배포 흐름을 문서화할 수 있을 때 도입 우선순위를 높인다.
- 다음 강점이 일정, 품질, 운영 비용 중 하나를 명확히 낮출 때 실무 채택 근거가 충분하다: 개인 업무 자동화와 단축키 구성에 빠르다.
- 다음 제약이 핵심 리스크라면 파일럿 구현, 성능 측정, 장애 대응 연습을 거친 뒤 본격 도입한다: 대규모 소프트웨어 개발에는 적합하지 않다.

## 도입 전 체크리스트

- 현재 팀이 이 언어의 빌드, 테스트, 디버깅, 배포 흐름을 문서화하고 반복 실행할 수 있는지 확인한다.
- 핵심 라이브러리와 프레임워크의 유지보수 상태, 라이선스, 보안 업데이트 흐름을 확인한다.
- 대체 언어와 비교해 학습 비용, 운영 리스크, 장기 인력 수급이 프로젝트 기간에 맞는지 확인한다.
- 공식 문서, 언어 스펙 또는 레퍼런스, 패키지/툴링 문서를 최소 1회 확인하고 프로젝트 표준 링크로 남긴다.

## 최신 확인 필요

- 인기 순위, 점유율, 채용 수요, 연봉 데이터는 답변 시점의 최신 자료로 확인한다.
- 최신 안정 버전, LTS 정책, 주요 프레임워크 버전은 공식 문서나 릴리스 노트로 확인한다.
- 보안 권고, 패키지 생태계 상태, 플랫폼 지원 종료 여부는 프로젝트 도입 전에 별도 확인한다.
