# WAT (WebAssembly Text Format)

원천: `docs/PROGRAMMING_LANGUAGES_2026.md`, 기준일 2026-05-13. 최신 순위와 시장 지표는 답변 시점에 웹으로 확인한다.

## 핵심 판단

WebAssembly 바이너리(WASM)의 사람이 읽을 수 있는 텍스트 표현이다. 직접 코딩보다 디버깅·교육·저수준 최적화 패치에 사용하는 도구적 언어다. WAT (WebAssembly Text Format) 도입 판단은 문법 선호보다 실행 환경, 기존 자산, 운영 인력의 숙련도에 더 크게 좌우된다. 제품의 주 경로가 WASM 바이너리 디버깅, 컴파일러 출력 검증, WASM 교육, 수동 최적화 마이크로 커널이고, 다음 제약을 통제할 수 있다면 우선 검토할 만하다: .

## 사용처

WASM 바이너리 디버깅, 컴파일러 출력 검증, WASM 교육, 수동 최적화 마이크로 커널. 실무에서는 이 범위를 넘어 모든 문제를 해결하는 범용 선택지로 보기보다, WAT (WebAssembly Text Format) 생태계가 이미 강한 플랫폼과 도구 체인에 맞춰 채택하는 편이 안정적이다. 신규 프로젝트에서는 장기 유지보수 인력과 배포 대상의 제약을 함께 확인해야 한다.

## 특징

S-표현식(Lisp 유사) 문법, 스택 기반 가상 머신 명령어 집합의 직독 표현, wat2wasm으로 바이너리 변환. 코드 작성 모델, 빌드 방식, 런타임 제약이 프로젝트 구조에 직접 영향을 주므로 초기 설계 단계에서 테스트 전략과 패키지 관리 방식을 같이 정해야 한다.

## 장점

. 특히 기존 생태계의 표준 도구를 그대로 활용할 수 있을 때 생산성과 운영 예측 가능성이 높다. 팀이 관용구와 디버깅 방식을 익히면 같은 기능을 더 적은 접착 코드로 구현할 수 있다.

## 제약

. 이 제약은 작은 실험에서는 드러나지 않다가 배포, 성능, 보안 감사, 장기 유지보수 단계에서 비용으로 나타나는 경우가 많다. 도입 전에는 대체 언어와 운영 모델을 같이 비교해야 한다.

## 적합한 프로젝트

- WASM 컴파일러 백엔드 개발 및 검증
- WASM 명세 학습과 런타임 동작 이해
- 크리티컬 패스에서 컴파일러 출력을 수동으로 튜닝하는 경우
- WAT (WebAssembly Text Format) 표준 도구와 런타임을 팀이 직접 운영할 수 있고, 코드 리뷰에서 언어별 관용구를 일관되게 적용할 수 있는 프로젝트.

## 부적합하거나 주의할 프로젝트

- 일반 웹 프론트엔드 — JavaScript/TypeScript가 훨씬 생산적
- 성능이 필요한 신규 프로젝트 — Rust/C로 작성 후 WASM 컴파일이 표준 경로
- 팀 전체에 WASM 내부 지식이 없는 경우
- WAT (WebAssembly Text Format) 경험자가 없는데 핵심 장애 대응, 성능 튜닝, 릴리스 자동화까지 동시에 요구되는 프로젝트.

## 대표 생태계와 도구

- `wat2wasm` / `wasm2wat` (WebAssembly Binary Toolkit, WABT)
- wasm-pack, wasm-bindgen (Rust ↔ WASM 브릿지)
- Emscripten (C/C++ → WASM)
- WASMtime, Wasmers (WASM 런타임)
- 브라우저 DevTools Sources 패널

## 학습 난이도와 선행 지식

어셈블리 언어 또는 스택 머신 개념이 있으면 빠르게 이해할 수 있다. 완전 입문자는 C/Rust → WASM 경로를 먼저 익히고 WAT는 보조 도구로 접근하는 것이 현실적이다. 입문 단계에서는 문법보다 작은 프로그램을 빌드, 테스트, 포맷, 배포하는 전체 흐름을 먼저 익히는 것이 좋다. 실무 투입 전에는 오류 처리, 의존성 관리, 표준 라이브러리 사용법을 팀 규칙으로 정리해야 한다.

## 운영/배포 관점

WAT 파일은 `wat2wasm`으로 `.wasm` 바이너리로 변환 후 브라우저, Node.js, 서버사이드 WASM 런타임에 배포. CDN 캐싱에 유리한 작은 바이너리 크기가 특징이다. 재현 가능한 빌드, 의존성 잠금, 런타임 버전 고정, 관측 로그 포맷을 초기에 정하지 않으면 운영 환경 차이로 인한 결함이 늘어난다.

## 타입/런타임 특성

정적 타입(i32/i64/f32/f64), 샌드박스 실행 환경, 선형 메모리 모델. GC/예외 Proposal 적용 시 타입 확장. 호스트 환경(브라우저/Node.js)과 임포트/익스포트로 통신. 이 특성은 API 경계, 테스트 범위, 병렬성 모델, 장애 격리 방식에 영향을 준다. 언어의 타입 시스템이 보장하지 않는 부분은 정적 분석, 테스트, 코드 리뷰 규칙으로 보완해야 한다.

## 실사용 예제

다음 예제는 이 언어를 단순 출력이 아니라 실제 업무 코드에서 자주 나오는 데이터 집계, 자동화, 런타임 제어, 또는 플랫폼 특화 작업에 적용하는 형태로 보여준다.

```wat
(module
  (memory (export "memory") 1)
  (func (export "add_paid")
    (param $current i32)
    (param $amount i32)
    (param $paid i32)
    (result i32)
    local.get $paid
    if (result i32)
      local.get $current
      local.get $amount
      i32.add
    else
      local.get $current
    end))
```
